PLAN FOR THIS PROJECT

WHAT ferrodb WILL HAVE + DESIGN CHOICES:
- relational db (just better)

Storage Engine:
- Will be disk based (easier for storing since it's not volatile like memory and more interesting)
- will use row oriented db (just better)
- probably heap organized data files (super fast point lookup + write, but has slow aggregate lookup)
- want to maximize sequential access (faster than random access)
- page size will be 4kb (maybe change)
- will track allocated pages using a bitmap (page_id = 0)
- once the bitmap page is full, then find a free page (make sure to either rewrite everything to 0s or only read up to the furthest allocated page), if not there, then allocate a new one specifically for that
- store a header on each bitmap page to point to the next one
- as more tuples are added, the slot array grows from top to bottom, and tuples grow from bottom up
- if a tuple is removed, "slide" remaining tuples to fill empty space (also update slot array pointer)
- use slotted pages (has header that has info on page and cells)
- slotted pages arent contigious (cuz pointers show where records are stored) so if there are "holes", you can insert data there (if it fits), or just add data to free space
- if there is enough free holes to create a row inside a page, then compact/defragment the page to free up space
- have to modify pointers to rows when shuffling stuff around too
- have page directory so it's easier to find pages/data files (synchronizing can be hard though)
- if there is ever data that is larger than a page, put it inside an overflow storage page
- serialize types using big-endian for fixed size
- Pascal String to serialize variable size data
- padding in tuples (easier for cpu to read, easier to modify values)
- IEEE 754 for floating point arithmetic
- for null data, you could store a bitmap in a header that says what is null (many different ways to do this)
- to insert a new tuple, check page directory for a page with a free slot, fetch page from disk to memory, check slot array if there is enough space in page (if not find another page/create one), insert tuple into page
- to update a tuple, find tuple using record id, if new value fits in the space, just update, if not, then mark old value as deleted and insert new value like it's a new tuple
- if data is deleted, you can nullify its pointer or remove it
- page directories will be stored on disk simlar to bitmap pages
- when a heap page is deallocated, num_entries decreases by 1, and everything is shifted left (ensures everything is contigious)

Buffer Pool:
- use memory for caching
- buffer pool has frames (same size as page) to put pages from disk into
- have a separate page table that has meta data about what pages are currently in memory
- pin counter will let the buffer pool know if a page is currently being used so it won't be evicted
- pin counter starts at 0 and increments by 1 for each reader/writeres and -1 when it's not being read
- when pin counter is 0, the page is no longer pinned and can be removed
- use latches on page tables when disk writes to the frame for thread safety (won't be overwriten)
- use ARC to evict frames, last time of accesses stored on metadata on page table
- ghost cache to keep track of last evicted (separate data structure)
- if there is a dirty page, buffer pool has to write back changes to disk so memory and disk have same data. do this periodically in the background
- once a dirty page is written, can evict or unset dirty flag on the page

Disk Manager:
- will use os functions since it's easier than using the raw disk
- each page will have a page ID, which can also be the file offset. it will be unique per table
- to read a page, it'll go to the offset which represents the pages id and writes the bytes into memory
- to write a page, it'll take the dirty page from memory and overwrite it
- you can also allocate a new page (a new page id will be created)/deallocate one

ARC cache:
- T1: pages accessed exactly once recently (recency)
- T2: pages accessed at least twice recently (frequency)
- B1: pages that have been evicted from T1
- B2: pages pages evicted from T2
- p: target size for T1, starts at 0, adapts over time, changes if some pages are accessed more than others
- if a page is already pinned, start from tail to head (from chosen list), if the first one is full, go to other one (O(n))
- if both are pinned, return None cuz buffer pool is full, just return an error (could add ability to retry later)

B+ Tree indexing:
- will use secondary key (a column(s)) to point to primary key so pointer updating is ez (you only have to update the primary key)
- b+ tree for indexing (has O(log N) lookup, where N is the number of items in the tree)
- traverse it by starting from root to leaf
- for the parent page id, instead of storing in header, i'll keep a path stack as i'm traversing down then using it to traverse back up
- root node page id will be stored in catalog (created later)
- Root node header: page_id, page_type, lsn, num_keys
- leaf node header have next and prev page ids for the sibling pointers
- Root node & internal nodes: |HEADER|<node1*>|key1|<node2*>|key2|...|keyn|<noden*>|(N keys and N+1 pointers), where node1 points to a child node where the key is less than k1,
node 2 points to a child node where the key is >= key1 and < key2 ... node n points to the child node who's key is >= key N
- Leaf nodes: |key|value|key|value| ... where the key is what we're searching for and the value is a pointer to the RecordId
- |HEADER|node array|key array| is better cuz they are loaded into memory contigiously and you can also just do binary search on the sorted keys
- the arrays will be fixed length (length should fit the page length perfectly so no wasted space, so depends on header, key, pointer size)
- leaf nodes will have sibling pointers from node to node, so you don't have to backtrack when searching and doing range scans (doubly linked list structure)
- index keys must be sorted (low -> high or high -> low)
- null keys will either be stored at start or end of each node
- Inserting: 
    1. find correct node, L 
    2. insert data entry into L in sorted order
    If L has enough space, we're done
    Else, we have to split the keys from L evenly to L and a new node, L2, copy middle key up to parent node (if leaf, if internal no copy) so it can point to the new page
    recursively do this process until the tree is balanced
    splitting root node can get complicated
- Deleting:
    1. find correct node, L where entry is
    2. remove entry
    If L is at least half full, done
    If L has m/2 -1 entries,
    try to redistribute, borrowing from sibling, then update the pointer in the parent at sibling to account for redistribution
    if that fails (sibling not more than half full), merge L and sibling
    recursively merge until tree is balanced
- for concurrency, use latch crabbing
- for duplicate keys, use composite keys: make the index key (indexed_value, primary key), making them unique since primary keys are unique
- for variable length keys, we can store it in a slotted page format
- Some optimizations are delaying merges to reduce disk io and then do them in a batch (log the changes to store them), maybe do these after

Architecture layers:
1. B+ trees and keys (does lookups)
2. Page layout and serialization (tuples + how they're endoed, slotted pages)
3. Buffer pool manager (frames, page table, pin/reference counters, ARC for eviction, ghost cache)
4. Disk manager (uses os functions to read/write/allocate/deallocate memory, page directory)
