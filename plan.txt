PLAN FOR THIS PROJECT

WHAT ferrodb WILL HAVE + DESIGN CHOICES:
- relational db (just better)

Storage Engine:
- Will be disk based (easier for storing since it's not volatile like memory and more interesting)
- will use row oriented db (just better)
- probably heap organized data files (super fast point lookup + write, but has slow aggregate lookup)
- want to maximize sequential access (faster than random access)
- page size will be 4kb (maybe change)
- will track allocated pages using a bitmap (page_id = 0)
- once the bitmap page is full, then find a free page (make sure to either rewrite everything to 0s or only read up to the furthest allocated page), if not there, then allocate a new one specifically for that
- store a header on each bitmap page to point to the next one
- as more tuples are added, the slot array grows from top to bottom, and tuples grow from bottom up
- if a tuple is removed, "slide" remaining tuples to fill empty space (also update slot array pointer)
- use slotted pages (has header that has info on page and cells)
- slotted pages arent contigious (cuz pointers show where records are stored) so if there are "holes", you can insert data there (if it fits), or just add data to free space
- if there is enough free holes to create a row inside a page, then compact/defragment the page to free up space
- have to modify pointers to rows when shuffling stuff around too
- have page directory so it's easier to find pages/data files (synchronizing can be hard though)
- if there is ever data that is larger than a page, put it inside an overflow storage page
- serialize types using big-endian for fixed size
- Pascal String to serialize variable size data
- padding in tuples (easier for cpu to read, easier to modify values)
- IEEE 754 for floating point arithmetic
- for null data, you could store a bitmap in a header that says what is null (many different ways to do this)
- to insert a new tuple, check page directory for a page with a free slot, fetch page from disk to memory, check slot array if there is enough space in page (if not find another page/create one), insert tuple into page
- to update a tuple, find tuple using record id, if new value fits in the space, just update, if not, then mark old value as deleted and insert new value like it's a new tuple
- if data is deleted, you can nullify its pointer or remove it
- page directories will be stored on disk simlar to bitmap pages
- when a heap page is deallocated, num_entries decreases by 1, and everything is shifted left (ensures everything is contigious)

Buffer Pool:
- use memory for caching
- buffer pool has frames (same size as page) to put pages from disk into
- have a separate page table that has meta data about what pages are currently in memory
- pin counter will let the buffer pool know if a page is currently being used so it won't be evicted
- pin counter starts at 0 and increments by 1 for each reader/writeres and -1 when it's not being read
- when pin counter is 0, the page is no longer pinned and can be removed
- use latches on page tables when disk writes to the frame for thread safety (won't be overwriten)
- use ARC to evict frames, last time of accesses stored on metadata on page table
- ghost cache to keep track of last evicted (separate data structure)
- if there is a dirty page, buffer pool has to write back changes to disk so memory and disk have same data. do this periodically in the background
- once a dirty page is written, can evict or unset dirty flag on the page

Disk Manager:
- will use os functions since it's easier than using the raw disk
- each page will have a page ID, which can also be the file offset. it will be unique per table
- to read a page, it'll go to the offset which represents the pages id and writes the bytes into memory
- to write a page, it'll take the dirty page from memory and overwrite it
- you can also allocate a new page (a new page id will be created)/deallocate one

ARC cache:
- T1: pages accessed exactly once recently (recency)
- T2: pages accessed at least twice recently (frequency)
- B1: pages that have been evicted from T1
- B2: pages pages evicted from T2
- p: target size for T1, starts at 0, adapts over time, changes if some pages are accessed more than others
- if a page is already pinned, start from tail to head (from chosen list), if the first one is full, go to other one (O(n))
- if both are pinned, return None cuz buffer pool is full, just return an error (could add ability to retry later)

B+ Tree indexing:
- will use secondary key (a column(s)) to point to primary key so pointer updating is ez (you only have to update the primary key)
- b+ tree for indexing (has O(log N) lookup, where N is the number of items in the tree)
- traverse it by starting from root to leaf
- for the parent page id, instead of storing in header, i'll keep a path stack as i'm traversing down then using it to traverse back up
- root node page id will be stored in catalog (created later)
- Root node header: page_id, page_type, lsn, num_keys
- leaf node header have next and prev page ids for the sibling pointers
- Root node & internal nodes: |HEADER|<node1*>|key1|<node2*>|key2|...|keyn|<noden*>|(N keys and N+1 pointers), where node1 points to a child node where the key is less than k1,
node 2 points to a child node where the key is >= key1 and < key2 ... node n points to the child node who's key is >= key N
- Leaf nodes: |key|value|key|value| ... where the key is what we're searching for and the value is a pointer to the RecordId
- |HEADER|node array|key array| is better cuz they are loaded into memory contigiously and you can also just do binary search on the sorted keys
- the arrays will be fixed length (length should fit the page length perfectly so no wasted space, so depends on header, key, pointer size)
- leaf nodes will have sibling pointers from node to node, so you don't have to backtrack when searching and doing range scans (doubly linked list structure)
- index keys must be sorted (low -> high or high -> low)
- null keys will either be stored at start or end of each node
- Inserting: 
    1. find correct node, L 
    2. insert data entry into L in sorted order
    If L has enough space, we're done
    Else, we have to split the keys from L evenly to L and a new node, L2, copy middle key up to parent node (if leaf, if internal no copy) so it can point to the new page
    recursively do this process until the tree is balanced
    splitting root node can get complicated
- Deleting:
    1. find correct node, L where entry is
    2. remove entry
    If L is at least half full, done
    If L has m/2 -1 entries,
    try to redistribute, borrowing from sibling, then update the pointer in the parent at sibling to account for redistribution
    if that fails (sibling not more than half full), merge L and sibling
    recursively merge until tree is balanced (do redistributing later (very hard bruh))
- for concurrency, use latch crabbing. ADD LATER ** (single threaded for now, add concurrency later)
- for duplicate keys, use composite keys: make the index key (indexed_value, primary key), making them unique since primary keys are unique
- for variable length keys, we can store it in a slotted page format
- for now it only supports fixed length keys *** (add variable length later)
- Some optimizations are delaying merges to reduce disk io and then do them in a batch (log the changes to store them), maybe do these after

Catalog:
- basically the databases metadata layer that has mapping of table names to schemas and storage locations
- reserved for page 1
- will track which columns are indexed, and where those structures are on disk
- stores schema info like column types, constraints, primary key, 
- when one catalog page gets full, it'll point to the next one like for bitmap pages
- when a root split happens, have to update it in catalog and write it to disk
- header: |page_type (1)|page_id (4)|next_catalog_page (4)|num_entries (2)|lsn (8)|checksum (4)|

SQL parsing:
- will use recursive descent since sql is recursive
- store token types in an enum
- separate error generating and error reporting code
- it will loop from the users input and return tokens until it reaches the end
- tokenizer is basically a switch case for all characters
- have tokens for string literals, comparison operators, punctuation (all in token enum)
- for keyword recognition, look up a sequence of characters in a table, if matches return keyword token, else return an identifier token
- keep scanning even if there is an error so the user knows about all of them
- for expressions like age > 2 AND name = 'bob' OR id < 3, this is the hierarchy (lowest binding to highest):
- OR -> AND -> NOT -> equality/comparison -> term -> factor -> unary -> primary
- parser will create an AST as it goes through the tokens
- abstract syntax tree is an enum of syntax types like Select { columns, table, where_clause}, Insert {table, values}, and expression enum 
BinaryOp {left, op, right}, Column(String), Literal (Value)
- when parser encounters an error, it will have to get to a state where the next token matches the rule being parsed (discard prev tokens)
- it will propagate an error through the recursive descent calls, then synchronize to discard tokens until a statement boundary
- Null is Value::Null, varchar(n) needs special handling cuz of length argument
- for now it'll support grammar like (case insensitive):
CREATE TABLE name (col type, ....)
INSERT INTO name VALUES (...)
SELECT cols FROM table WHERE expr
UPDATE table SET col = val WHERE expr
DELETE FROM table WHERE expr
CREATE INDEX index_name ON table (col)
- no:
JOIN, subqueries, aggregates, GROUP BY, ORDER BY, yet

Query execution engine:
- iterator (volcano) model (most space efficient, simple)
- everything wrapped in a driver loop (runs the whole tree)
- tuple stream is basically "open, next, close" (Exceutor trait)
- pull based (much simpler than push)
- can seqscan + filter & index scan (optimizer chooses which later)
- sequential scan uses heapscanner
- have to construct plan on the statement tree before doing operations
- also have to fetch heap, schema, and indexes using get_table(name)
- for secondary indexes, lookups take two jumps
- insert gets tuples from VALUES expressions -> Vec<Value> -> serialize tuple -> heap insert (with checking said below)
- expression evaluation used for filter, projections output columns, values for insert/update
- modification operators have to maintain indexes (and update secondary indexes), and check constaints
- to solve halloween problem, keep track of modified record ids for each query
    future optimizations:
    - compile to optimize it (like JIT in postgres)
    - multi index scans to find overlap in AND statements 
    - zone map: pre compute aggregations for tuples. then db decides if it wants to access page, so it can skip irrelevant tuples
    - zone map will store values like: min, max, avg, sum, count, num nulls, etc ...

Binder: 
- sits between parser and planner
- creates a scope: ordered list of in scope tables with their qualifiers (alias/name) and combined output schema
- does name resolution (verifies table names in catalog, column names in scope) and alias/qualifier resolution (flags duplicate aliases)
- star expansion: resolves star into columns
- further checks if is valid sql semantics: type checking, insert arity, etc. (uses DataType)
- emits bound tree where names are replaced with indexes
- will use global column index to catch column naming errors
- will emit a logical plan for the optimizer to turn into physical operators + a bound output schema for row layout, column names

Query optimization:
- logical plan goes into optimizer, which uses catalog to get schema info + statistics 
- optimizer also consults cost model which has stuff like magic constants to determine best path
- then optimizer creates physical plan (which tells executor how to execute query)
- no matter how "weird" a query might be the optimizer should still output a reasonable plan
- optimizer can kep track of things like whether a col has an index, or if the data is sorted (for ORDER BY, merge join), etc. so 
- Selinger bottom up DP for join ordering
- as you traverse down you keep track of best plan you've seen so far, if one path has higher cost, eliminate
- will only take a single query and create a physical plan for that (not multiple)
- rule and cost based aren't mutually exclusive, both 
- Rule based (easy but relies on magic constants and impossible to give good plans for complex queries):
    - optimizer will be able to permute operators in the logical plan , then determine wether one is better than the other (must be equivalent)
    - also have transformations: logical op -> physical op or  logical -> logical that we think will lead to a better result
    - generate alternatives via transformation that follow relational algebra equivalences
    - can switch around filter (one is more selective than other), break down predicates into conjunctive clauses, then move around
    commutative/associative joins (only INNER). never do cross joins unless forced
    - push down predicate/limit/projection
    - many many many possible rules to apply
    - search algorithm to look over possible enumerations, what transformations to apply/not apply, when to terminate (exhausted, time/cost threshold/transformation count (best)), 
- Cost based:
    - cost model makes an educated guess on the "cost" of a query plan 
    - predict cpu cycles, how much i/o (and seq vs random), ram consumption. depends heavily on hardware & magic constants
    - estimate output size per operator (relies on statistics on db)
    - column and table stats
    - statistics will be stored in memory for simplicity
    - they're emited by every operator (output cardinality + (V and min/max) per column)
    - if no statists, fallback to magic constants
    - store: N_R (number of tuples in R), V(A,R) (num distinct values for attribute A), min/max
    - selection cardinality is used to determine the number of tuples that will be selected for a given output (too many to put here)
    - use selectivity formulas for computing estimated cost for different operations 
    - clamp selectivity to [0, 1], floor cardinality at 1
    - equi-depth histograms so buckets have roughly same amount of elements
    - cardinality estimations need to be good cuz errors compound
    - selectivity assumes: uniform data, independent predicates, containment principle (join keys will overlap)

WAL + crash recovery:
- use steal + no force policies for better runtime performance
- ARIES protocol for crash recovery
- write to a log (in memory) before writing log to disk
- log must be written to disk before dirty pages are written to disk
- <BEGIN> to mark starting point 
- will record: transaction id, object id, before value, after value, checksum
- when transaction is done, dbms appends <COMMIT> to the log
- if db crashes, we have the WAL on disk so we can just follow that to restore
- group commit optimization to batch multiple log flushes together (use a timeout)
- also flush when log buffer is full
- use physiological logging (log has page id and slot number)
- redo all txns, then undo ones that didn't commit
- will write checkpoints every 512mb of data written to the log
- can also delete logs before a checkpoint
- each data page contains a pageLSN (updated when txn modifies page), the system keeps track of a flushedLSN, the max LSN flushed so far
- flushedLSN acts as a boundary where disk buffer stops and in memory log continues (updated in memory every time we write the WAL buffer to disk)
- before a page is flushed to disk, its pageLSN <= flushedLSN. if not, the flushedLSN hasn't "caught up" to that page yet
- if the buffer pool wants to evict a certain page but the pageLSN hasn't caught up, we can flush the other logs to disk and then evict that page
- recLSN to record the last update to a page since it was last flushed (stored on dirty page table)
- lastLSN to record teh latest record of a txn (stored in active transaction table)
- master record (global LSN) is stored on disk and stores the LSN of latest checkpoint
- when a txn commits, we write a <COMMIT> record to log and guarantees all log records up to <COMMIT> are flushed to disk
- when commit succeeds, write TXN-END to log, indicating that no new log record that that txn will appear in the log ever again, does not need to be immediately flushed either,
but once flushed, we can "trim" the log until the TXN-END record
- if a txn ever aborts, we use the CLR to undo the changes 
- CLR contains fields of a normal log + undo_nextLSN pointer (next to be undone LSN)
- we don't need to wait for CLR to be flushed before notifying the application that the txn aborted
- abort algorithm: 
    - write ABORT log
    - go through the txn's updates in reverse order. for each one:
        - write a CLR entry to the log
        - restore old Value
    write TXN-END log and release locks
- ATT (active transaction table):
    - one entry per currently active txn (txn_id, status (running, commit, undo entry), lastLSN)
    - once TXN-END, can remove its entry
- DPT (dirty page table):
    - tracks dirty pages in the bp once the checkpoint starts
    - each page has a rec_LSN (oldest record since it was last written to disk so we know whether the page was modified before or after checkpoint)
- use fuzzy checkpointing: we allow dbms to keep running while the system writes records for a checkpoint, now we don't have to flush dirty pages
- ARIES recovery algorithm:
    - 1. analysis
        - start from last checkpoint
        - if TXN-END, remove its transaction from ATT
        - for all other transactions, add entry to ATT with status UNDO and on commit, change to COMMIT
        - for update records, if a page P isn't in the DPT, then add to DPT and set P's recLSN to log records's LSN
        - at the end, ATT identifies which txns were active at time of crash and DPT identifies dirty pages that might've not been flushed to disk
    - 2. redo
        - scan foward starting from smallest recLSN in DPT
        - for each record or CLR with a LSN:
            - redo action unless one of the following is true:
            - target page isn't in DPT
            - if target page is in DPT, but the record's LSN < page's recLSN. dbms doesn't need to retrive page from disk for this check
            - if target page is in DPT, but the log record's LSN <= pageLSN. need to retrive page from disk for this check
            - reapply logged update, set pageLSN to log record's LSN
    - 3. undo txns that were active at time of crash 
        - for each txn in reverse LSN order (using lastLSN):
            - pick largest lastLSN across all transactions in the ATT
            - traverse lastLSN's in the same order, but in reverse for how the updates happened originally
            - write a CLR for the modification
